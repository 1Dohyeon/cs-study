# DNS (Domain Name System)

사람은 `google.com`을 기억하지만, 컴퓨터는 `142.250.196.110` 같은 IP 주소로 통신한다. 이 둘을 연결해주는 것이 **DNS**다. 인터넷의 전화번호부라고 보면 된다.

## DNS 조회 과정

브라우저에 `www.example.com`을 입력하면 어떤 일이 벌어질까?

```
브라우저
  │
  ├─ 1. 브라우저 캐시 확인
  ├─ 2. OS 캐시 확인 (hosts 파일 포함)
  ├─ 3. 로컬 DNS 서버(Resolver)에 질의
  │       │
  │       ├─ 4. 루트 네임서버에 질의  →  ".com 담당은 여기"
  │       ├─ 5. TLD 네임서버에 질의   →  "example.com 담당은 여기"
  │       └─ 6. 권한 네임서버에 질의  →  "www.example.com = 93.184.216.34"
  │
  └─ 7. IP 주소 반환 → TCP 연결 시작
```

캐시에 있으면 바로 반환하고, 없으면 단계적으로 네임서버를 찾아간다.

## 재귀 질의 vs 반복 질의

DNS 조회 방식은 두 가지다.

| 방식 | 동작 | 누가 발품을 파나 |
|---|---|:---:|
| 재귀 질의 (Recursive) | Resolver가 최종 답을 찾아서 클라이언트에게 반환 | Resolver |
| 반복 질의 (Iterative) | 서버가 "다음엔 여기 물어봐"를 반환, 클라이언트가 직접 따라감 | 클라이언트 |

실제로는 클라이언트 → Resolver 구간은 **재귀**, Resolver → 네임서버 구간은 **반복** 방식으로 동작한다.

## 주요 DNS 레코드 타입

| 레코드 | 역할 |
|:---:|---|
| A | 도메인 → IPv4 주소 |
| AAAA | 도메인 → IPv6 주소 |
| CNAME | 도메인 → 다른 도메인 (별칭) |
| MX | 메일 서버 지정 |
| NS | 도메인의 네임서버 지정 |

## TTL과 캐시

DNS 응답에는 TTL(Time To Live) 값이 포함된다. 이 시간 동안은 캐시된 값을 그대로 쓰고, 만료되면 다시 질의한다.

- TTL이 짧으면: 변경사항이 빠르게 반영되지만 질의가 잦아짐
- TTL이 길면: 질의 횟수가 줄지만 IP 변경 시 전파가 느림

## 요약

- DNS는 도메인 이름 → IP 주소 변환 시스템
- 조회 순서: 브라우저 캐시 → OS 캐시 → Resolver → 루트 → TLD → 권한 네임서버
- 클라이언트↔Resolver는 재귀 질의, Resolver↔네임서버는 반복 질의
- TTL로 캐시 유효 시간을 제어 — 짧으면 유연, 길면 효율적
