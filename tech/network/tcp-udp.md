# TCP vs UDP

데이터를 전송하는 방법은 두 가지다. 확실하게 보내거나, 빠르게 보내거나. 이 둘을 담당하는 것이 전송 계층의 **TCP**와 **UDP**다.

## TCP (Transmission Control Protocol)

TCP는 데이터가 **반드시 도착함**을 보장한다. 그 대신 연결을 맺고 끊는 과정이 필요하다.

### 3-way Handshake (연결 수립)

```
클라이언트                          서버
    |  ── SYN ──────────────────→  |   "연결할게"
    |  ←─────────────── SYN-ACK ── |   "알겠어, 나도 연결할게"
    |  ── ACK ──────────────────→  |   "확인"
```

- **SYN:** 연결 요청
- **SYN-ACK:** 요청 수락 + 서버도 연결 요청
- **ACK:** 수락 확인

### 4-way Handshake (연결 종료)

```
클라이언트                          서버
    |  ── FIN ──────────────────→  |   "끊을게"
    |  ←─────────────────── ACK ── |   "알겠어"
    |  ←─────────────────── FIN ── |   "나도 끊을게"
    |  ── ACK ──────────────────→  |   "확인"
```

종료는 4단계인 이유: 서버가 아직 보낼 데이터가 남아 있을 수 있기 때문에 FIN과 ACK을 분리한다.

### TCP의 신뢰성 보장 메커니즘

| 기능 | 설명 |
|---|---|
| 순서 보장 | 패킷에 번호를 붙여 순서대로 재조립 |
| 오류 감지 | 손상된 패킷은 재전송 요청 |
| 흐름 제어 | 수신 측 버퍼 크기에 맞게 전송 속도 조절 (슬라이딩 윈도우) |
| 혼잡 제어 | 네트워크 상태에 따라 전송량 조절 |

## UDP (User Datagram Protocol)

UDP는 연결 수립 없이 데이터를 그냥 던진다. 도착 여부, 순서, 오류 복구 — 모두 보장하지 않는다. 대신 그만큼 빠르다.

"보낸다"는 행위 자체가 목적일 때 쓴다. 스트리밍, 게임, DNS 조회처럼 약간의 손실보다 **지연이 더 치명적인** 상황에서 선택한다.

## 비교

| 항목 | TCP | UDP |
|---|:---:|:---:|
| 연결 방식 | 연결형 (3-way handshake) | 비연결형 |
| 신뢰성 | 보장 | 보장 안 함 |
| 순서 보장 | O | X |
| 속도 | 상대적으로 느림 | 빠름 |
| 헤더 크기 | 20바이트 | 8바이트 |
| 사용 예 | HTTP, 이메일, 파일 전송 | DNS, 스트리밍, 게임, VoIP |

## 요약

- TCP는 신뢰성을 위해 연결 수립(3-way)과 종료(4-way) 과정을 거침
- UDP는 연결 없이 빠르게 전송, 손실을 감수
- 흐름 제어는 수신 측 과부하 방지, 혼잡 제어는 네트워크 과부하 방지
- 어떤 것을 쓸지는 "신뢰성 vs 속도" 트레이드오프로 결정
