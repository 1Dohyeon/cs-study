# HTTP 기초

브라우저에 URL을 입력하면 무슨 일이 일어날까? 그 과정의 핵심이 **HTTP(HyperText Transfer Protocol)**다. 클라이언트와 서버가 데이터를 주고받는 규칙이다.

## HTTP의 특징

| 특징 | 설명 |
|---|---|
| 비연결성 (Connectionless) | 요청에 응답한 뒤 연결을 끊음 |
| 무상태성 (Stateless) | 이전 요청을 기억하지 않음 |
| 텍스트 기반 | 요청·응답 모두 사람이 읽을 수 있는 텍스트 형식 |

무상태성 때문에 로그인 상태 같은 정보는 HTTP 자체가 아니라 쿠키·세션·토큰으로 별도 관리한다.

## 요청 메시지 구조

```
GET /index.html HTTP/1.1          ← 요청 라인: 메서드 / 경로 / 버전
Host: www.example.com             ← 헤더
Accept: text/html
                                  ← 빈 줄 (헤더 끝)
                                  ← 바디 (GET은 보통 없음)
```

### HTTP 메서드

| 메서드 | 용도 | 멱등성 |
|---|---|:---:|
| GET | 리소스 조회 | O |
| POST | 리소스 생성 | X |
| PUT | 리소스 전체 수정 | O |
| PATCH | 리소스 일부 수정 | X |
| DELETE | 리소스 삭제 | O |

> **멱등성:** 같은 요청을 여러 번 보내도 결과가 동일한 성질. GET을 100번 보내도 서버 상태는 변하지 않는다.

PUT과 PATCH는 둘 다 수정 메서드인데 멱등성이 다르다. 차이는 **무엇을 보내느냐**에 있다.

- **PUT**은 리소스 전체를 덮어쓴다. `{ name: "kim", age: 20 }`을 PUT으로 10번 보내면 결과는 항상 같다 → 멱등
- **PATCH**는 일부만 수정한다. 그런데 `{ age: age + 1 }`처럼 현재 값 기준으로 변경하는 요청이면 보낼 때마다 결과가 달라진다 → 비멱등

PATCH 자체가 항상 비멱등인 건 아니다. `{ age: 20 }`처럼 절댓값으로 지정하면 멱등하게 동작한다. 다만 **설계에 따라 멱등성이 보장되지 않을 수 있기 때문에** 비멱등으로 분류한다.

## 응답 메시지 구조

```
HTTP/1.1 200 OK                   ← 상태 라인: 버전 / 상태코드 / 메시지
Content-Type: text/html           ← 헤더
Content-Length: 1024

<html>...</html>                  ← 바디
```

### 주요 상태 코드

| 코드 | 의미 |
|:---:|---|
| 200 | OK — 요청 성공 |
| 201 | Created — 리소스 생성 성공 |
| 301 | Moved Permanently — 영구 리다이렉트 |
| 302 | Found — 임시 리다이렉트 |
| 400 | Bad Request — 잘못된 요청 |
| 401 | Unauthorized — 인증 필요 |
| 403 | Forbidden — 권한 없음 |
| 404 | Not Found — 리소스 없음 |
| 500 | Internal Server Error — 서버 오류 |
| 503 | Service Unavailable — 서버 일시 불가 |

## HTTPS

HTTP에 **TLS(Transport Layer Security)** 암호화를 더한 것이 HTTPS다.

```
클라이언트                          서버
    |  ── ClientHello ──────────→  |   지원하는 암호화 방식 전달
    |  ←────────── ServerHello ──  |   선택한 방식 + 인증서 전달
    |  (인증서 검증)                |
    |  ── 대칭키 교환 ───────────→  |
    |  ←──────────────── 완료 ───  |
    |  ══ 암호화 통신 시작 ════════ |
```

- 인증서는 CA(인증 기관)가 서버의 신원을 보증
- 키 교환 이후엔 대칭키로 빠르게 통신

## 요약

- HTTP는 비연결·무상태 프로토콜 — 상태 관리는 별도 수단 필요
- 요청: 메서드 + 경로 + 헤더 + 바디 / 응답: 상태코드 + 헤더 + 바디
- 멱등성 — GET·PUT·DELETE는 O, POST·PATCH는 X
- HTTPS = HTTP + TLS 암호화, 인증서로 서버 신원 보증
