# HTTP 버전별 비교

HTTP는 웹의 성장과 함께 계속 진화했다. 버전마다 이전 버전의 병목을 어떻게 해결했는지를 중심으로 보면 흐름이 보인다.

## HTTP/1.1

1997년 등장. 현재도 널리 사용되는 버전이다.

**특징**
- **Keep-Alive:** 기본적으로 연결을 유지해 매 요청마다 TCP 연결을 새로 맺지 않음
- **파이프라이닝:** 응답을 기다리지 않고 여러 요청을 연속으로 전송 가능

**문제 — HOL Blocking (Head-of-Line Blocking)**

요청을 순서대로 처리하기 때문에, 앞 요청이 느리면 뒤 요청이 모두 막힌다. 파이프라이닝이 있어도 응답은 요청 순서대로만 돌아온다.

## HTTP/2

2015년 등장. HOL Blocking을 해결하는 것이 핵심 목표였다.

**핵심 개선**

| 기능 | 설명 |
|---|---|
| 멀티플렉싱 | 하나의 TCP 연결에서 여러 요청·응답을 동시에, 순서 무관하게 처리 |
| 헤더 압축 (HPACK) | 반복되는 헤더를 압축해 전송량 감소 |
| 서버 푸시 | 클라이언트가 요청하기 전에 서버가 필요한 리소스를 미리 전송 |
| 바이너리 프레이밍 | 텍스트 대신 바이너리로 파싱 효율 향상 |

**여전한 문제**

TCP 자체의 HOL Blocking은 남아있다. 패킷 하나가 손실되면 TCP 레벨에서 전체 스트림이 대기한다.

## HTTP/3

2022년 공식 표준. TCP를 버리고 **QUIC** 프로토콜로 교체했다.

**QUIC이란?**
- UDP 기반으로 만든 새로운 전송 프로토콜
- 연결 수립과 TLS 핸드셰이크를 동시에 처리 → 연결 지연 감소
- 스트림별 독립적인 패킷 손실 처리 → TCP의 HOL Blocking 완전 해결

```
HTTP/1.1: 요청 → 응답 → 요청 → 응답  (순차)
HTTP/2:   요청1 ──────────────────→
          요청2 ──────────────────→  (동시, 단 TCP 위)
          요청3 ──────────────────→

HTTP/3:   요청1 ──────────────────→
          요청2 ──────────────────→  (동시, QUIC 위 — 패킷 손실도 독립 처리)
          요청3 ──────────────────→
```

## 버전 비교

| 항목 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|:---:|:---:|:---:|
| 전송 프로토콜 | TCP | TCP | QUIC (UDP 기반) |
| 멀티플렉싱 | X | O | O |
| 헤더 압축 | X | O (HPACK) | O (QPACK) |
| HOL Blocking | 있음 | TCP 레벨 존재 | 해결 |
| TLS | 선택 | 사실상 필수 | 기본 내장 |

## 요약

- HTTP/1.1 → HTTP/2: 멀티플렉싱·헤더 압축으로 HOL Blocking 완화, 단 TCP 한계 존재
- HTTP/2 → HTTP/3: TCP를 QUIC으로 교체해 전송 계층 HOL Blocking까지 해결
- HTTP/3은 연결 수립 속도도 빨라져 모바일·불안정 네트워크에서 유리
